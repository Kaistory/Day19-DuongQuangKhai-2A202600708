Kỹ thuật lập chỉ mục Vector
(Vector Indexing / ANN)

Từ brute-force đến HNSW: cách đánh đổi giữa tốc độ, độ chính xác (recall) và bộ nhớ để tìm top-k vector gần nhất trong hàng triệu điểm.

← → để chuyển slide · phím F để fullscreen

Tại sao cần "lập chỉ mục"?

Tìm kiếm vector = tìm k điểm gần nhất (k-NN) với query embedding.

  • Exact k-NN (brute-force): so sánh query với mọi vector → O(N·d).
  • 1M vector × 768 chiều = quá chậm cho real-time (P99 > 100ms).
  • Giải pháp: ANN — Approximate Nearest Neighbor. Hy sinh một chút recall để nhanh hơn 10–100×.

Index = cấu trúc dữ liệu giúp bỏ qua phần lớn vector không liên quan.

Tam giác đánh đổi
Recall (chính xác) /\ / \ / \ / \ Speed /________\ Memory (độ trễ) (RAM/đĩa) Mỗi thuật toán = một điểm trong tam giác này.

Bản đồ các họ thuật toán

1 · Flat

Brute-force / Exact

Quét tuyến tính. Recall 100%, không cần "train". Baseline để đo các index khác.

2 · Tree

KD-Tree, Annoy, Ball-Tree

Chia không gian đệ quy. Tốt ở chiều thấp; "lời nguyền số chiều" ở d cao.

3 · Hashing

LSH

Băm vector gần nhau vào cùng bucket. Đơn giản, streaming-friendly.

4 · Quantization

PQ, OPQ, SQ

Nén vector để tiết kiệm RAM. Thường ghép với IVF.

5 · Inverted

IVF (clustering)

Gom cụm, chỉ tìm trong vài cụm gần query.

6 · Graph

HNSW, NSG, DiskANN

Đồ thị "small-world". SOTA về speed/recall — Qdrant dùng cái này.

1 · Flat Index Brute-force

So sánh query với toàn bộ N vector, sắp xếp theo khoảng cách.

  • ✔ Recall = 100% (ground truth chuẩn).
  • ✔ Không build/train, không tham số tinh chỉnh.
  • ✗ Chậm tuyến tính: không hợp > vài trăm nghìn vector.

Dùng khi: N nhỏ (< 50k), cần độ chính xác tuyệt đối, hoặc làm baseline đo recall cho ANN.

Độ phức tạp
Chỉ sốGiá trị
QueryO(N·d)
BuildO(1)
MemoryO(N·d) — full vector
Recall100%

2 · IVF Inverted File Index

Dùng k-means để chia không gian thành nlist cụm (Voronoi cells). Mỗi vector thuộc về cụm gần nhất.

Lúc query: chỉ tìm trong nprobe cụm gần query nhất → bỏ qua phần còn lại.

  • nprobe ↑ → recall ↑ nhưng chậm hơn (tham số then chốt).
  • Cần "train" trên dữ liệu mẫu để học các centroid.
  • Biến thể: IVF-PQ (nén) cho dataset tỉ vector.
┌─────┬─────┬─────┐ │ · │ ·· ◎│ · │ ◎ = query ├─────┼─────┼─────┤ ● = cụm được quét │ ·●· │ ·●· │ · │ (nprobe = 2) ├─────┼─────┼─────┤ │ · │ · │ · │ └─────┴─────┴─────┘ nlist cụm → chỉ quét nprobe cụm

3 · Product Quantization PQ — nén vector

Cắt vector d-chiều thành m đoạn nhỏ, mỗi đoạn lượng tử hóa thành 1 byte (codebook 256 centroid).

  • 768 float32 = 3072 byte → PQ m=96 = 96 byte (giảm ~32×).
  • Khoảng cách tính xấp xỉ qua bảng tra (ADC) — rất nhanh.
  • OPQ: xoay không gian trước khi nén để giảm sai số.
  • SQ (Scalar Quant): float32 → int8, đơn giản hơn, giảm 4×.

PQ hiếm khi đứng một mình — ghép với IVF (IVF-PQ) hoặc HNSW (HNSW-PQ).

Đánh đổi

✔ Tiết kiệm RAM khổng lồ → chứa tỉ vector trong bộ nhớ.

✗ Mất mát thông tin → recall giảm; cần re-rank top-k bằng vector gốc để bù lại.

OPQ & IVF-PQ ⭐ combo cho dữ liệu tỉ vector

OPQ — Optimized PQ

Xoay rồi mới nén

PQ giả định các đoạn (sub-vector) độc lập — sai khi các chiều tương quan. OPQ học một ma trận xoay R sao cho sau khi xoay, dữ liệu phân bố đều giữa các đoạn → sai số lượng tử nhỏ hơn.

  • ✔ Cùng số byte, recall cao hơn PQ thường.
  • Chỉ tốn thêm 1 phép nhân ma trận lúc encode/query.
IVF-PQ — gom cụm + nén

IVF chọn vùng · PQ nén vector

IVF giới hạn tìm kiếm trong nprobe cụm; PQ nén vector trong cụm xuống vài byte → vừa nhanh vừa nhẹ RAM.

  • Pipeline FAISS kinh điển: OPQ → IVF → PQ.
  • Lưu residual (vector − centroid) rồi mới PQ → chính xác hơn.
  • Re-rank top-k bằng vector gốc để bù mất mát.
# FAISS — factory string gói cả 3 kỹ thuật
index = faiss.index_factory(384, "OPQ96,IVF4096,PQ96")
#                            └OPQ┘ └─IVF─┘ └PQ┘
index.train(xb); index.add(xb)
index.nprobe = 16            # số cụm quét lúc query (recall ↔ tốc độ)

4 · LSH Locality-Sensitive Hashing

Thiết kế hàm băm sao cho vector gần nhau có xác suất cao rơi vào cùng bucket.

  • Query → băm → chỉ so sánh các vector cùng bucket.
  • Dùng nhiều bảng băm (L tables) để tăng recall.
  • ✔ Streaming/insert dễ, lý thuyết đảm bảo rõ ràng.
  • ✗ Thường thua HNSW/IVF về recall-vs-speed thực tế → ít dùng làm index chính ngày nay.

5 · Tree-based Annoy / KD-Tree

Chia nhỏ không gian bằng siêu phẳng (Annoy: chia ngẫu nhiên), tạo nhiều cây → query đi xuống lá.

  • Annoy (Spotify): forest of random projection trees, file mmap, đọc từ đĩa tốt.
  • ✔ Build nhanh, index bất biến dễ share giữa process.
  • ✗ "Curse of dimensionality": ở d cao (>100) hiệu quả giảm, khó update incremental.

Phổ biến trong recommender; HNSW thường thắng cho semantic search.

6 · HNSW ⭐ Qdrant dùng mặc định

Hierarchical Navigable Small World — đồ thị nhiều tầng. Tầng trên thưa (nhảy xa), tầng dưới dày (tinh chỉnh). Greedy search từ trên xuống.

  • ✔ SOTA: recall cao + latency thấp, không cần train.
  • ✔ Insert động (thêm vector không build lại).
  • ✗ Ngốn RAM (lưu graph + vector); build chậm hơn IVF.
L2: ◎───────────● (thưa) \ / L1: ◎──●────●──●──● (vừa) \ \ / / L0: ●─●─●─●─●─●─●─●─●─● (đầy đủ) bắt đầu tầng cao → tụt dần → top-k

HNSW — 3 tham số phải nhớ

Tham sốÝ nghĩaTăng lên thì…
mSố cạnh / node (bậc đồ thị) recall ↑, RAM ↑, build chậm
ef_constructĐộ rộng tìm kiếm lúc build chất lượng graph ↑, build chậm
ef (hnsw_ef)Độ rộng tìm kiếm lúc query recall ↑, latency ↑
# Qdrant — cấu hình HNSW khi tạo collection
from qdrant_client.models import VectorParams, Distance, HnswConfigDiff

client.create_collection(
    "docs",
    vectors_config=VectorParams(size=384, distance=Distance.COSINE),
    hnsw_config=HnswConfigDiff(m=16, ef_construct=100),  # build-time
)
# query-time: chỉnh ef để đổi recall ↔ tốc độ
client.search("docs", query_vector=q, limit=10,
              search_params={"hnsw_ef": 128})

Chọn index thế nào? Cây quyết định

N < 50k ? ───── có ──► Flat (exact, recall 100%) │ không ▼ RAM thoải mái & cần recall cao? ── có ──► HNSW ⭐ (mặc định nên chọn) │ không ▼ Dataset cực lớn (tỉ vector), RAM hạn chế? ── có ──► IVF-PQ / DiskANN │ không ▼ Cần insert streaming đơn giản? ── có ──► LSH / IVF

Qdrant (lab này) mặc định HNSW; có thể bật quantization (SQ/PQ) để giảm RAM, và indexing_threshold để hoãn build trên collection nhỏ.

  • Index: Qdrant tự dựng HNSW mỗi collection.
  • Hybrid search: dense (HNSW) + sparse/BM25 → fuse.

Thử nghiệm: đổi hnsw_ef và xem Precision@10 ↔ P99 thay đổi ra sao.

Việc cần làm
  • Tăng ef → recall lên, latency lên bao nhiêu?
  • Bật quantization → RAM giảm, recall mất bao nhiêu?
  • So Flat vs HNSW trên cùng tập truy vấn.
Tóm tắt

Index = đánh đổi có chủ đích

Flat cho đúng tuyệt đối · IVF/PQ cho quy mô & RAM · HNSW cho cân bằng tốt nhất.

Hiểu 3 trục Recall — Speed — Memory, rồi tinh chỉnh tham số (nprobe / ef / m) theo SLA của bạn.
